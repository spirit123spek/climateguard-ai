# Supabase Schema Design

This document translates the ClimateGuard AI database architecture into a Supabase-ready schema design. It defines the planned PostgreSQL tables, columns, relationships, ownership model, storage buckets, indexing approach, and migration readiness requirements.

This is a design document only. It does not contain SQL, migrations, policies, or created Supabase resources.

## Schema Principles

1. Store only real observed climate data, imported historical data, user actions, calculated metrics, or model outputs.
2. Keep observed climate readings separate from user simulation inputs.
3. Preserve source, timestamp, unit, model version, and calculation metadata for traceability.
4. Use Supabase Auth as the identity provider and `users` as the application profile table.
5. Protect all user-owned data with Row Level Security.
6. Restrict external data ingestion, prediction writes, report generation, and audit writes to trusted backend services.

## Table: users

Application profile table linked to Supabase Auth.

| Column | Data Type | Nullable | Constraints | Relationships |
| --- | --- | --- | --- | --- |
| `id` | UUID | No | Primary key, generated by database | Referenced by `predictions.user_id`, `simulations.user_id`, `reports.user_id`, `alert_history.user_id`, `audit_logs.actor_user_id` |
| `auth_user_id` | UUID | No | Unique, must map to one Supabase Auth user | References `auth.users.id` |
| `full_name` | Text | Yes | Maximum length should be defined before migration | None |
| `email` | Text | No | Unique or unique per active account, must be normalized | Mirrors Supabase Auth email for lookup and display |
| `role` | Text | No | Allowed values should include `user`, `analyst`, `admin`; default should be `user` | Used by RLS and admin workflows |
| `default_city` | Text | Yes | Must not be used as climate data source by itself | Used for dashboard query defaults |
| `default_country` | Text | Yes | Optional profile preference | Used with `default_city` |
| `preferences` | JSONB | No | Default empty object | Stores units, notification preferences, dashboard preferences |
| `created_at` | Timestamp with time zone | No | Set on creation | Audit and sorting |
| `updated_at` | Timestamp with time zone | No | Updated on profile changes | Audit and sorting |

## Row Level Security Strategy

| Table | RLS Strategy |
| --- | --- |
| `users` | Users can read and update only their own profile where `auth_user_id` matches the authenticated Supabase user. Admin access requires explicit role checks. |
| `climate_readings` | Authenticated users can read validated shared readings. Inserts, updates, and deletes are restricted to backend service role or ingestion jobs. |
| `predictions` | Users can read predictions they own through `user_id`. Shared city predictions require a separate read policy. Writes are restricted to backend ML services. |
| `simulations` | Users can create, read, update labels for, and delete their own simulations subject to retention rules. They cannot access other users' simulations. |
| `reports` | Users can read and request only their own reports. Status updates and storage path writes are restricted to backend report services. |
| `alert_history` | Users can read and acknowledge their own user-specific alerts. Shared city alerts require controlled read policy. Creation is restricted to backend alert services. |
| `audit_logs` | General users should not access audit logs directly. Admins and backend services may access restricted records. Writes should be append-only through backend services. |

## Authentication Ownership Model

Supabase Auth owns authentication identity. The `users.auth_user_id` column links each application profile to `auth.users.id`.

User-owned records use `user_id` foreign keys pointing to `users.id`. RLS policies should resolve the authenticated Supabase user to the matching application profile before allowing access to simulations, reports, predictions, user-specific alerts, and profile data.

System-owned records such as shared climate readings, ingestion audit entries, and shared city predictions are written by backend services using secure server-side credentials. These records are not owned by individual users unless a `user_id` is explicitly present.

## User Data Isolation Model

1. Users can access only their own profile, simulations, reports, user-requested predictions, and user-specific alerts.
2. Users may read approved shared climate readings because readings are environmental records, not personal records.
3. Simulation inputs are private to the user who created them.
4. Report files are private to the report owner.
5. Audit logs are restricted to service roles and authorized administrators.
6. Future organization or team access must introduce explicit organization ownership instead of weakening user-level isolation.

## Storage Buckets Needed

| Bucket | Purpose | Access Model | Retention |
| --- | --- | --- | --- |
| `reports` | Generated PDF reports and report exports | Private bucket; users access only files referenced by their own `reports` rows through signed URLs or backend-mediated download | Retain while `reports.status` is completed and before `expires_at`, unless policy requires longer |
| `source-payloads` | Optional raw API or dataset payload snapshots for debugging and provenance | Private bucket; service-role access only by default | Shorter retention than normalized records unless required for audit or model reproducibility |
| `model-artifacts` | Optional trained model files, encoders, feature metadata, and evaluation artifacts | Private bucket; backend ML service access only | Retain active model versions and required historical model versions |

## Report Storage Strategy

Generated reports should be stored as files in the private `reports` bucket. The `reports` table stores ownership, status, report type, city, source date range, `source_summary`, `storage_bucket`, `storage_path`, and lifecycle timestamps.

Report generation should compile only stored climate readings, prediction outputs, simulations, and calculated metrics with documented provenance. The frontend should request report downloads through a backend endpoint or signed URL workflow that verifies the authenticated user owns the report metadata row.

## File Retention Strategy

| File Type | Retention Strategy |
| --- | --- |
| Generated reports | Retain until `expires_at` if expiration is enabled; retain metadata longer than the file for traceability |
| Raw source payloads | Retain temporarily for ingestion debugging, source verification, and model reproducibility; delete or archive according to source and compliance requirements |
| Model artifacts | Retain active production model and previous versions needed to reproduce stored predictions |

| Table | Required Indexes | Query Purpose |
| --- | --- | --- |
| `users` | `auth_user_id`, `email`, `role` | Auth profile lookup, account lookup, admin filtering |
| `climate_readings` | `city`, `country`, `recorded_at`, `ingested_at`, `source`, `source_type`, `quality_status` | Dashboard loading, time-series trends, freshness checks, source validation |
| `predictions` | `climate_reading_id`, `user_id`, `simulation_id`, `predicted_at`, `risk_category`, `alert_index` | Latest prediction lookup, user prediction history, alert generation |
| `simulations` | `user_id`, `baseline_climate_reading_id`, `city`, `created_at`, `predicted_risk_category` | Simulator history, city scenarios, high-risk scenarios |
| `reports` | `user_id`, `report_type`, `city`, `status`, `created_at`, `generated_at`, `expires_at` | User report lists, report queue, retention jobs |
| `alert_history` | `user_id`, `prediction_id`, `climate_reading_id`, `city`, `alert_index`, `delivery_status`, `created_at` | Alert timelines, delivery status, severe alert filtering |
| `audit_logs` | `actor_user_id`, `actor_type`, `action`, `entity_type`, `entity_id`, `created_at`, `city` | Security review, entity history, ingestion and ML job investigation |

Composite index candidates:

* `climate_readings(city, recorded_at)` for dashboard time-series queries.
* `climate_readings(source_type, city, recorded_at)` for live versus historical analytics.
* `climate_readings(source, ingested_at)` for ingestion quality checks.
* `predictions(user_id, predicted_at)` for user prediction history.
* `predictions(climate_reading_id, predicted_at)` for reading-based prediction lookup.
* `simulations(user_id, created_at)` for simulator history.
* `reports(user_id, created_at)` for report lists.
* `reports(status, created_at)` for report generation workers.
* `alert_history(user_id, created_at)` for user alert timelines.
* `audit_logs(entity_type, entity_id)` for entity-level audit trails.

## Query Optimization Strategy

1. Store frequently filtered values as scalar columns instead of only inside JSONB.
2. Use JSONB for flexible metadata, source details, feature snapshots, and calculation provenance.
3. Query dashboards by `city` and recent `recorded_at` windows before joining prediction records.
4. Prefer latest validated reading queries filtered by `quality_status = valid`.
5. Avoid loading raw source payloads during dashboard views.
6. Use report metadata rows to list reports; load files only when the user requests a download.
7. Keep prediction history queries scoped by `user_id`, `city`, or `climate_reading_id`.
8. Consider future daily or monthly aggregate tables only after real query volume proves they are needed.

## Data Lifecycle Classification

| Data | Stored Permanently | Cached | Regenerated | Notes |
| --- | --- | --- | --- | --- |
| User profiles | Yes, while account exists | No | No | Deletion policy must follow account lifecycle |
| Live climate readings | Retained long term or archived after high-resolution retention period | Recent dashboard responses may be cached by backend | No | Observed data should not be regenerated |
| Historical dataset readings | Yes | No | Reimport possible from source if dataset version is preserved | Important for training and trend baselines |
| Predictions | Yes or archived after retention period | Latest city prediction may be cached | Yes, only from original inputs and model version when artifact exists | Stored predictions preserve exact displayed result |
| ESG and sustainability scores | Stored with predictions | Latest score may be cached | Yes, if formula version and inputs are preserved | No hardcoded scores |
| Simulation inputs | Yes, user-owned until deletion or retention policy | No | No | Inputs are user-generated scenario data |
| Simulation results | Yes, with simulation history | No | Yes, if model and formula versions are preserved | Must remain tied to baseline reading |
| Alerts | Yes for alert history retention period | Active alert status may be cached | Yes from prediction history if rules are preserved | User acknowledgement must not be regenerated |
| Reports metadata | Yes, longer than file retention | No | No | Metadata proves report source and ownership |
| Report files | Retained per file policy | Download URLs are temporary | Yes, if source records still exist | Regenerated file may differ in formatting if template changes |
| Audit logs | Yes for audit retention period | No | No | Append-only security record |

## Migration Readiness Checklist

Before writing actual SQL migrations, confirm the following:

1. Final allowed values for roles, source names, source types, quality statuses, risk categories, alert indexes, report statuses, delivery statuses, and audit actor types.
2. Numeric scales and valid ranges for AQI, PM2.5, PM10, temperature, humidity, rainfall, wind speed, ESG score, sustainability score, risk score, and confidence score.
3. Exact unit standards for stored weather and pollutant values.
4. Supabase Auth profile creation flow and whether `users` rows are created by trigger, backend service, or explicit API call.
5. RLS policies for every table, including admin access and service-role-only writes.
6. Storage bucket names, privacy settings, signed URL strategy, and report download flow.
7. Retention periods for live readings, raw source payloads, predictions, simulations, reports, alerts, and audit logs.
8. Whether shared city predictions are public to authenticated users or always user-scoped.
9. Whether historical data remains in `climate_readings` or moves to specialized historical tables after scale review.
10. Report source reference strategy: JSONB `source_summary` only or future join tables for report-to-reading and report-to-prediction links.
11. Audit log append-only enforcement approach.
12. Initial indexing set and expected dashboard query patterns.
13. Backup, restore, and environment separation plan for development, staging, and production Supabase projects.
14. Model artifact versioning plan so stored predictions can be explained or reproduced.
15. Confirmation that no fake, hardcoded, placeholder, or demo climate values will be inserted by migrations or seed scripts.
